# 🕵️‍♂️ Adversarial Audit & Refinements

Acest document compilează rezultatele auditului critic realizat de cei 6 sub-agenți de testare și deciziile adoptate pentru remedierea vulnerabilităților și îmbunătățirea calității inginerești a aplicației Meridian Onboarding.

---

## 1. 🏛️ Arhitectură & Securitate (Architecture)
*   **Probleme identificate:** SPA client-side offline fără server. Datele sunt izolate în IndexedDB-ul fiecărui utilizator (fără sursă unică de adevăr). Drepturile de Admin pot fi escaladate prin modificarea variabilelor în React DevTools.
*   **Decizii adoptate:**
    *   Migrăm de la arhitectura exclusiv client-side (React + IndexedDB) la o arhitectură full-stack: React frontend (Cloudflare Pages) + FastAPI backend (Railway) + PostgreSQL (Neon).
    *   Arhitectura urmează principiile SOLID cu pattern-ul Layered + Repository + Domain Events.
    *   Securizarea autentificării se mută de la Web Crypto API client-side la JWT server-side cu bcrypt.
    *   Pre-boarding lock verificat server-side via JWT claims (`hireDate`).
    *   Integritatea referențială se rezolvă nativ cu FK-uri PostgreSQL și `CASCADE DELETE`.
    *   Domain Events (`EmployeeDeleted` → `CascadeDeleteHandler` + `BuddyOrphanHandler`) pentru side-effects cross-domain.
    *   Documentăm arhitectura țintă și limitele demo-ului (JWT simplu fără OAuth2, single-instance fără HA) în [[ASSUMPTIONS]] și [[DECISIONS]].

## 2. 📚 Biblioteci și Framework-uri (Libraries & Frameworks)
*   **Probleme identificate:** Virtualizarea listei de 210 angajați cu `@tanstack/react-virtual` este overengineering, strică accesibilitatea (a11y) și blochează căutarea `Ctrl+F`. Validările din UI sunt dublate și nu reutilizează schemele Zod.
*   **Decizii adoptate (revizuite pentru full-stack):**
    *   **Eliminăm `@tanstack/react-virtual`** — cu backend-ul FastAPI, lista de angajați se paginează server-side (`GET /employees?page=1&limit=20`). Frontend-ul primește maxim 20-50 de elemente per pagină, eliminând orice nevoie de virtualizare. Se redobândește `Ctrl+F`, accesibilitatea și simplitatea codului.
    *   **Adăugăm `@tanstack/react-query`** (TanStack Query) — înlocuiește `DbContext` monolitic. Fiecare hook (`useEmployees`, `useChecklist`, `useScheduler`) gestionează independent caching, invalidare, optimistic updates și refetch automat.
    *   **Validarea devine pe două straturi:** Pydantic (server-side) este sursa unică de adevăr pentru validare. Zod rămâne **exclusiv pe client** pentru UX instant (feedback vizual pe câmpuri de formular înainte de submit). Nu mai există duplicare — schemele Zod pe client oglindesc contractul API.
    *   **Eliminăm `localforage`** ca dependență de producție — toată persistența se mută pe PostgreSQL. `localforage` se poate păstra doar pentru cache offline opțional.

## 3. 💾 Structura Bazei de Date (Database Structure)
*   **Probleme identificate:** Abuz de key-value prin stocarea de monoliți JSON (array-uri mari de date). Complexitate $O(N)$ (Read-Modify-Write) asincronă expusă la Race Conditions (pierderi de date la scrieri concurente). Lipsa constrângerilor de bază de date (Foreign Keys).
*   **Decizii adoptate (TOATE problemele rezolvate prin migrarea la PostgreSQL):**
    *   **Tabele relaționale** în loc de monoliți JSON: `employees`, `checklist_tasks`, `schedule_entries`, `tips` — fiecare cu indecși proprii și coloane tipizate.
    *   **FK-uri native + `ON DELETE CASCADE`**: Ștergerea unui angajat curăță automat la nivel de DB toate checklist-urile și programările asociate. Zero cod custom în Service Layer pentru cascade.
    *   **Tranzacții ACID**: Race conditions eliminate — operațiunile concurente sunt serializate de PostgreSQL. Nu mai există Read-Modify-Write asincron.
    *   **Constrângeri `UNIQUE`**: Unicitatea `assignedDesk` aplicată la nivel de schemă DB (`UNIQUE(desk_id)`), nu în logica aplicației.
    *   **Migrații cu Alembic**: Orice modificare de schemă este versionată, reversibilă și reproductibilă.

## 4. 🗃️ Modelul de Date (Data Model)
*   **Probleme identificate:** Proprietăți de dată reprezentate ca string-uri simple fără format, lipsă format strict pe ID-uri, enums amestecate lingvistic, lipsă validări condiționale în Zod pentru stări corelate (ex. un task `skipped` fără `skipReason`).
*   **Decizii adoptate (revizuite pentru PostgreSQL + Pydantic):**
    *   **Date calendaristice**: Coloane SQLAlchemy `Date` și `DateTime` native — PostgreSQL gestionează intern formatul, eliminând parsing-ul de string-uri. Pydantic serializează automat ca ISO 8601.
    *   **ID-uri**: UUID v4 generate de PostgreSQL (`gen_random_uuid()`) ca primary key pe fiecare tabel. Elimină ID-urile secvențiale predictibile (`emp-4`, `task-1`).
    *   **Enum-uri standardizate**: Toate enum-urile din DB în limba engleză (`pending`, `in_progress`, `completed`, `skipped`, `blocked`; `OFFICE`, `REMOTE`, `HYBRID`). Traducerea se face exclusiv în layer-ul de prezentare (React).
    *   **Validări condiționale**: Pydantic `@model_validator` (echivalentul server-side al `superRefine`) impune: `skip_reason` obligatoriu doar când `status == 'skipped'`; `blocked_by` obligatoriu doar când `status == 'blocked'`. Backend-ul respinge cu 422 orice payload invalid.

## 5. 🔄 Fluxurile Utilizatorilor (User Workflows)
*   **Probleme identificate:** Blocarea totală a scheduler-ului în pre-boarding împiedică angajatul să-și planifice prima zi. Seeding-ul generează sarcini tehnice identice pentru toate departamentele. Logica de skip generează deadlock (task-ul blocat de un task sărit nu se deblochează niciodată). Importul JSON este destructiv fără confirmare.
*   **Decizii adoptate (revizuite — logica se mută pe backend):**
    *   **Pre-boarding scheduler**: Backend-ul verifică JWT claim-ul `hireDate`. Dacă `hireDate > now()`, permite programare **doar pentru săptămâna de start** (hireDate → hireDate + 4 zile). Endpoint-ul `PUT /scheduler` respinge cu 403 orice zi în afara ferestrei permise.
    *   **Checklist-uri per departament**: `ChecklistService.get_template_for_dept(department)` returnează task-uri specifice: Engineering (IDE, Git, PR), Sales (CRM, scripts), Marketing (Brand Book, analytics), Finance (facturare, compliance), HR (admin tools). Template-urile sunt stocate ca seed data în PostgreSQL.
    *   **Rezolvare deadlock la skip**: `ChecklistService.skip_task(task_id)` implementează o mașină de stări: când un task părinte este omis, backend-ul parcurge automat arborele de dependențe și deblochează toate task-urile copil (`status: 'blocked'` → `status: 'pending'`). Logica e atomică într-o singură tranzacție PostgreSQL.
    *   **Import JSON cu preview diff**: Endpoint-ul `POST /backup/preview` primește JSON-ul, îl validează cu Pydantic, compară cu starea curentă din DB și returnează un diff structurat (adăugări, modificări, ștergeri). Frontend-ul afișează diff-ul într-un dialog de confirmare. Doar după confirmare se apelează `POST /backup/restore`.

## 6. 🎨 Decizii UI/UX & Accesibilitate (A11y)
*   **Probleme identificate:** Rapoarte de contrast WCAG sub limită (Cyan pe Alb este 2.1:1, text pe Navy este 2.68:1). Ecusonul QR nu are etichete ARIA. Mecanismul de copiere Slack ("clipboard hijacking") este invaziv și nu anunță cititoarele de ecran. Simulation date are text microscopic.
*   **Decizii adoptate (revizuite):**
    *   **Contrast WCAG AA (4.5:1 minim)**: Înlocuim Cyan `#2BC4D9` cu o nuanță mai închisă (`#0E8A9A`) pe fundal alb. Text pe Navy devine alb pur (`#FFFFFF`) sau gri deschis (`#D1D5DB`). Verificăm fiecare combinație cu un tool de contrast automat.
    *   **Accesibilitate QR Badge**: Adăugăm `role="img"` și `aria-label="Security badge QR code for [Employee Name]"` pe container-ul QR. Adăugăm `alt` descriptiv pe imaginea ecusonului.
    *   **Clipboard cu anunț vocal**: La copierea unui șablon Slack, adăugăm un `<div aria-live="polite" class="sr-only">` care anunță cititoarelor de ecran: *"Slack message template copied to clipboard"*. Eliminăm copierea automată — acțiunea se declanșează exclusiv la click pe butonul dedicat.
    *   **Cele 3 tipuri de Skip Task se PĂSTREAZĂ** (Inline, Modal, Drawer) — sunt cerute explicit de testele Playwright din [[1. Onboarding Checklist]]. Nu le uniformizăm. Standardizăm în schimb logica internă: toate cele 3 UI-uri apelează același endpoint backend `PATCH /checklists/:empId/tasks/:taskId/skip` cu `{ skipReason: string }`. Diferența rămâne exclusiv vizuală.
    *   **Simulation Date**: Mărim font-ul selectorului de dată din bara de simulare la minim `14px` și adăugăm `aria-label="Simulation date selector"` pentru accesibilitate.
